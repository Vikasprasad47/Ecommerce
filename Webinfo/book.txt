3.1.1 Registration (Email Method)
✔ USER FLOW (Step-by-Step)

User opens /register.

Enters:

Full Name

Email

Password

Frontend validates:

Email format

Password strength (zxcvbn)

Sends request to backend:
POST /api/user/register

Backend:

Sanitizes inputs (Mongo, XSS, etc.)

Validates using Yup

Checks if email already exists

Hashes password (bcryptjs with salt)

Saves user

Generates OTP for email verification

Sends email using Nodemailer/Resend

User receives OTP

User enters OTP on /Otp-Verification

Backend matches OTP → marks email as verified

Backend generates:

Access Token (short lifespan)

Refresh Token (long lifespan)

Tokens stored in httpOnly secure cookies

User is automatically logged in

Redux updates user state

User is redirected to home/dashboard

3.1.2 Registration — TECHNICAL DEEP DIVE
🔵 Frontend Techniques

React Hook Form for form management

Yup resolver for validation

Debounced email validation

Password strength meter using zxcvbn

Axios with interceptors

Suspense + lazy loading

Toasts for feedback

🔵 Backend Techniques

Rate limiting (authLimiter) to prevent brute-force

bcryptjs hashing

Nodemailer with SMTP or Resend API

OTP expiration logic (5–10 min)

MongoDB indexes on email

Sanitization using:

express-mongo-sanitize

xss-clean

validator library

🔵 Security

No password stored in plain text

No OTP stored in plain text (hashed OTP recommended)

httpOnly cookies for tokens

CSRF-resistant architecture

UUID for verification request

🔵 Data Model Important Fields

✔ 4.2.1 Middleware Stack (In Order)

Order matters. Quickoo's backend loads middleware in a strategic sequence:

1. CORS
app.use(cors({
  credentials: true,
  origin: process.env.FRONTED_URL,
}));


Why?

Prevents unauthorized origins.

Allows cookies to be sent across requests.

Ensures secure domain–API communication.

2. express.json()

Parses incoming JSON bodies with:

limit checks (optional)

sanitization compatibility

3. cookieParser()

Allows:

reading JWT tokens

handling refresh sessions

secure cookie operations

4. morgan('common')

Logging strategy for:

request method

path

status

latency

error events

Essential for debugging and production logs.

5. helmet()

Protects from:

cross-site scripting

MIME sniffing

clickjacking

embedding attacks

Used with:

helmet({ crossOriginEmbedderPolicy: false })


Because Cloudinary + some frontend components require disabling COEP.

6. Body Parser (optional)

Used when necessary.

7. CORS + Security + Auth Together

This order guarantees:

tokens visible to server

API protected

headers correct

cookies parsed

Key Attributes (Descriptive Only)

Full Name
The user’s real name or account name.

Email Address
Must be unique. Used for login, notifications, invoices, and verification.

Phone Number
Optional for email login users; required for phone login users.

Password Hash
Securely hashed representation of user password.
Never stored directly.

Profile Avatar URL
Cloudinary-hosted image.

Account Verification Status
Indicates if the user has verified their email or phone number.

User Role
Defines platform permissions:
“user”, “seller”, “admin”.

Account Status
Can be: active, banned, disabled, under-review.

Date Fields (Creation & Update)
For audit trail, analytics.

Refresh Token Record
Stored for managing long-term sessions.

Behavioral Rules

A user cannot access seller routes unless role = seller.

Admin privilege overrides all restrictions.

Multiple failed login attempts will trigger rate limiters.

When account is deleted, personal data retention follows privacy policy rules.

⭐ 26.3 How Inventory Changes in Real-Time
✔ When seller uploads product → stock added
✔ When customer adds to cart → stock unaffected

(Cart does not reserve inventory)

✔ When order is placed → stock RESERVED

reservedStock increases.

✔ When seller accepts order → reservedStock stays same

(it’s already reserved)

✔ When seller cancels order → reservedStock decreases

Stock returns to available.

✔ When customer completes pickup → totalStock decreases

reservedStock is cleared.

✔ When seller updates stock → system recalculates instantly
⭐ 26.4 The Inventory Reservation System (Core Engine)

This prevents overselling.

Why reservations?

If 10 people try to buy the last 1 item:

only 1 should get it

others receive “Out of stock” cleanly

What triggers reservation?

Placing an order.

Not:

adding to cart

viewing product

pre-checkout

Reservation expiry rules:

If:

payment fails

seller cancels

order times out

→ reservation removed.
